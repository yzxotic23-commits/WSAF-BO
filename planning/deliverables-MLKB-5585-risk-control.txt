1. Operational Findings Summary
   Written summary of validation results from live FeedFlow PRO runs, including:

   - AI provider load (Codex rate limit) under parallel multi-pair feeding
   Each finding tagged: Blocker | Major | Minor.

2. Account Consistency Validation Checklist
   Pass/Fail checklist covering AMS account slots vs FeedFlow pairs:
   - PAIR_COUNT in .env matches UI pair count
   - All accounts linked before Start feeding (N/N linked)
   - Each pair maps to correct account slots (pair 1 = account1+2, pair 2 = account3+4, etc.)
   - Profile/display names consistent between sidebar, chat preview, and feeding log
   - Partner-only message routing — no cross-pair message bleed
   - Auth/session folders exist per account slot after link
   Tested on Windows with 2-pair baseline and 3+ pair scenario.

3. Risk Control Matrix
   Table of operational risks with control, owner, and severity:
   - Same IP / shared proxy within a pair → bot messages fail to arrive
   - Insufficient proxies for account count → pair 3+ falls back to direct
   - New pair without manual warm-up → inbound ignored (strict partner filter)
   - Parallel AI calls exceed Codex limit → pair 3+ silent / fallback only
   - Wrong Windows build installed → app does not open or unstable behaviour
   - Data loss on update → sessions/settings wiped (mitigated in v1.0.48+)
   Each row: Risk | Likelihood | Impact | Current Control | Gap | Action.

4. Proxy & IP Governance Standard
   Short operational rule set for AMS/FeedFlow runs:
   - Minimum 1 unique working proxy per account (3 pairs = 6 proxies)
   - Never run bot-to-bot feeding with both accounts on same IP
   - Verify [PROXY] Assignment summary in System log before feeding
   - Warn / block Start feeding when pair shares proxy or uses direct fallback
   - Document direct-connection exception criteria (dev/test only).

5. Multi-Pair Feeding Validation Report
   Evidence from controlled test runs:
   - 2-pair run: opener, AI reply, nudge, completion — Pass/Fail
   - 3+ pair run: each pair receives opener and at least one AI round-trip — Pass/Fail
   - System log excerpts per pair ([Pair 1], [Pair 2], [Pair 3] …)
   - Screenshot of linked accounts + feeding log for failed pairs
   - Note on manual warm-up requirement (1 test message between new pair accounts).

6. Preventive Governance Mechanisms
   List of preventive controls to adopt in AMS operations:
   - Pre-feeding gate: all linked + proxy check + PAIR_COUNT sync
   - Version policy: approved release list (e.g. avoid v1.0.22–1.0.32; track latest stable)
   - New pair onboarding SOP: link → manual ping → verify log → start feeding
   - Post-feeding review: audit log + pair result summary per run
   - Escalation path when pair stops or ignored inbound warnings appear.
